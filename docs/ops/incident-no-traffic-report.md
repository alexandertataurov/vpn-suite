## ЭТАП 2 — Локализация

### Ветка 1: Handshake отсутствует

На текущем сервере для peer `GcQY...` handshake **есть** и transfer растёт. Если у вас на клиенте «latest handshake» пустой:
- Проверить, что в приложении выбран конфиг с этим же PrivateKey (и значит pubkey `GcQY...`).
- Endpoint в конфиге: `185.139.228.171:45790` — порт совпадает с listen port контейнера.
- Если подключаетесь с другого сервера/порта — нужна диагностика там.

### Ветка 2: Handshake есть, но трафика нет (на клиенте)

На сервере трафик по этому peer идёт (rx/tx). Если на клиенте трафика нет:
1. **MTU** — попробовать в конфиге клиента `MTU = 1280` (или 1380/1420), переподключиться.
2. **DNS** — в конфиге указаны 1.1.1.1, 1.0.0.1; если сайты не открываются, но `ping 8.8.8.8` работает — проблема DNS (UDP 53 через туннель или блокировка DoH).
3. **Маршруты на клиенте** — AllowedIPs = 0.0.0.0/0, ::/0 (full-tunnel); убедиться, что на клиенте нет правил, отправляющих трафик мимо туннеля.

## ЭТАП 3 — Синхронизация admin-api ↔ amnezia-wg (persist / reconcile)

### Текущее поведение

- **Source of truth:** БД (devices с server_id, не revoked/suspended).
- **Apply:** При issue (и при sync-peer) вызывается `add_peer` (docker exec `wg set ... peer ... allowed-ips ... preshared-key ...`). После apply вызывается `ensure_reply_routes` (ip route add 10.8.1.0/24 dev awg0 при необходимости).
- **Reconcile loop** (NODE_DISCOVERY=docker): каждые ~60 с сверяются пиры из БД с `wg show`; недостающие — добавляются, лишние (нет в БД) — удаляются, при дрифте allowed_ips/psk — update (remove + add). При старте admin-api выполняется **однократный** `reconcile_all_nodes()`, затем цикл.
- **Persist:** Пиры **не** записываются в конфиг-файл ноды; после рестарта контейнера amnezia-wg все пиры пропадают и восстанавливаются при следующем reconcile (в течение ~60 с или сразу при старте admin-api).

### Рекомендации (чтобы не ломалось снова)

1. **Два устройства с одним 10.8.1.2/32**  
   Отозвать лишний девайс или перевыпустить с другим адресом, чтобы в БД на один server_id был только один активный peer с 10.8.1.2/32.

2. **Reconcile уже есть**  
   Метрики: `vpn_peers_expected`, `vpn_peers_present`; алерт VpnPeersDrift. При старте admin-api выполняется начальный reconcile — пиры после рестарта ноды поднимаются без ручного вмешательства (в течение цикла).

3. **Опционально: persist пиров на ноде**  
   Чтобы не зависеть от времени до первого reconcile после рестарта контейнера:
   - либо при каждом изменении пиров записывать полный конфиг интерфейса и при старте контейнера применять его через `wg syncconf` (если amneziawg поддерживает);
   - либо скрипт PostUp в контейнере, который при старте запрашивает список пиров у admin-api и выполняет `wg set` для каждого (сложнее, нужен API/токен).

4. **Идемпотентность**  
   Повторный `add_peer` с теми же данными уже не ломает конфиг (wg set перезаписывает peer).

## ЭТАП 4 — Что сделать сейчас

### 1) Краткий вывод

- **Почему могло казаться, что «нет трафика»:** на сервере для peer 10.8.1.2 (GcQY...) handshake и трафик есть; возможны MTU/DNS на клиенте или выбор другого конфига/устройства. Дополнительно: в БД два устройства с одним и тем же 10.8.1.2/32 — один из них лишний.
- **Синхронизация:** reconcile работает; после рестарта ноды пиры восстанавливаются при следующем цикле или при старте admin-api; постоянного сохранения пиров в конфиг ноды нет (опционально — см. выше).
