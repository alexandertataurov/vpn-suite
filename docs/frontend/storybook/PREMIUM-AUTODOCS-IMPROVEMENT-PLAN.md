# Premium Autodocs Improvement Plan

A production-grade plan to evolve Storybook Autodocs into a premium design system documentation experience for designers, engineers, QA, and product teams.

---

## 1. Executive Summary

### What is currently wrong with default-looking Autodocs

- **Prop-dump feel**: Autodocs reads like an API catalog—Description → Canvas → ArgsTable—with no decision support or usage guidance.
- **Generic layout**: Default Storybook typography, spacing, and framing feel developer-only; no visual hierarchy or premium presentation.
- **Weak information architecture**: Raw autogenerated order (description, stories, props) does not guide "start here" or "when to use."
- **Missing decision support**: No Do/Don't, known limitations, related components, status badges, or copy-paste recipes.
- **Identical feel per component**: Every component page looks the same; no category-specific emphasis (buttons vs dialogs vs tables).
- **Toy demos**: Stories often use placeholder data; interactive flows (focus trap, keyboard, empty states) are under-demonstrated.
- **Maintenance friction**: Admin requires manual `parameters.docs.propsTable.rows`; Miniapp lacks Usage/A11y tabs and structured sections.

### Target experience

Autodocs should feel like a **curated internal design system product**:

- **Decision tool**: Clear "use this when" / "don't use when" / "use X instead of Y."
- **Design reference**: Variants, states, composition examples, content guidelines.
- **QA surface**: Realistic demos, edge cases, accessibility expectations.
- **Usage guide**: Start-here summary, recipes, related components, known limitations.
- **Premium presentation**: Intentional layout, spacing, typography, framed previews, scannable tables.

## 2. Audit of Current Weaknesses

### Visual weaknesses

| Area | Admin (SB 8) | Miniapp (SB 10) |
|------|--------------|-----------------|
| Max width | 72rem (post-upgrade) | 72rem |
| Section spacing | `--ds-docs-section-gap` | Present |
| Preview framing | `.docblock-preview` with label | Present |
| ArgsTable | Custom PropsTable (manual rows) | `.docblock-argstable` styled |
| Code blocks | Styled with language label | Styled |
| Callouts | Callout (tokens) | DocsCallout |
| TOC | Yes | No |
| Empty tabs | Generic fallback text | N/A (single scroll) |

**Gaps**: No "Start here" block; Do/Don't only in Admin Usage tab (manual); no status badge in Miniapp hero; no Known Limitations or Recipe blocks.

### IA/content weaknesses

- **Admin**: Tabs (Overview/Props/Usage/A11y) require manual `parameters.docs.usage`, `parameters.docs.propsTable.rows`, `parameters.docs.accessibility`. Empty tabs show fallback text.
- **Miniapp**: Single-scroll flow (Description → Primary → Canvas → ArgTypes). No Usage, A11y, or Do/Don't sections.
- **Both**: No structured "Start here," "When to use," "Known limitations," "Recipes," or "Related components" in default flow.
- **Both**: `description.component` supports `##` headings but authors rarely structure it; no enforcement.

### Interactivity weaknesses

- **Guided demos**: No play-function demos or guided state switching.
- **Context shells**: Some stories use `InContext` but many are isolated.
- **Edge cases**: Empty, loading, error, focus, keyboard stories exist but are not surfaced as "critical states."
- **Responsive**: Viewport toolbar exists (Miniapp) but no responsive composition examples.
- **Realistic data**: Admin uses `.storybook/data`; Miniapp mixes real and mock; no consistent "product-like" standard.

### Scaling/maintainability weaknesses

- **Admin PropsTable**: Manual `parameters.docs.propsTable.rows`—no auto-extraction from argTypes.
- **Admin Usage/A11y**: Manual `parameters.docs.usage`, `parameters.docs.accessibility`—high per-component overhead.
- **Miniapp**: No Usage or A11y sections at all.
- **Component-specific docs**: No `parameters.docs.sections` or category-based composition; one structure fits all.
- **Governance**: STORY-DOCUMENTATION-LAYERS defines conventions but no automated checks.

## 3. Global Improvements

| Improvement | Why it matters | Mechanism | Impact |
|-------------|----------------|-----------|--------|
| **Start-here block** | First thing users see; reduces "where do I begin?" | `parameters.docs.startHere` (string or node) → render above Description | High |
| **Status badge everywhere** | Maturity/risk at a glance | `parameters.status` or `parameters.docs.hero.status` → ComponentHero | Medium |
| **Do/Don't in default flow** | Reduces misuse; design-system standard | `parameters.docs.usage` → UsagePanel in Overview (Admin) / Description flow (Miniapp) | High |
| **Known limitations block** | Sets expectations; reduces support | `parameters.docs.limitations` (string[]) → reusable block | Medium |
| **Related components block** | Improves discoverability | `parameters.docs.related` (links[]) → reusable block | Medium |
| **Recipe block** | Copy-paste patterns for common cases | `parameters.docs.recipes` (title, code, description) → CodeBlock + caption | High |
| **ArgsTable auto-extraction** | Removes manual PropsTable rows | When `parameters.docs.propsTable.rows` absent, derive from argTypes | High |
| **Category grouping in ArgsTable** | Easier scan; Advanced vs primary | Use `argTypes[].table.category` → group or dim Advanced | Medium |
| **Preview frame with label** | Curated demo feel | Already in docs.css; ensure all previews use it | Done |
| **Consistent typography** | Premium feel | docs.css already uses tokens; verify prose width | Done |
| **Callout in MDX** | Best practices, warnings, a11y | Callout/DocsCallout in docs.components | Done |

### Recommended Storybook mechanisms

- **DocsContainer**: Theme + docs.page (already custom).
- **parameters.docs**: `startHere`, `usage`, `limitations`, `related`, `recipes`, `sections` (optional).
- **parameters.docs.components**: Override block components (Callout, etc.).
- **docs.css**: Global typography, spacing, tables, preview framing, callouts.
- **argTypes.table.category**: For ArgsTable grouping.

## 4. Per-Component Improvements

| Category | Emphasize | Standard sections | Optional sections |
|----------|-----------|-------------------|-------------------|
| **Button** | Hierarchy, destructive use, loading, icon | Start here, Variants, States, Do/Don't, API | Recipes (primary CTA, danger), Content guidelines |

| **Dialog** | Trigger flow, focus trap, destructive confirm | Start here, Flow, Variants, A11y, Do/Don't, API | Sizing, Layered interaction |
| **Table/DataGrid** | Density, overflow, empty/loading/error | Start here, Variants, States, Do/Don't, API | Sorting, Sticky, Content stress |
| **Tabs** | Active state, overflow, content | Start here, Variants, States, API | Label length, Accessibility |
| **Tooltip/Popover** | Trigger, timing, positioning | Start here, Variants, A11y, API | Collision, Accessibility |
| **Card** | Content hierarchy, layout, actions | Start here, Variants, Composition, API | Media, Action placement |
| **Badge/Tag** | Semantic usage, truncation | Start here, Variants, Do/Don't, API | Removable, Density |
| **Alert/Toast** | Severity, stacking, dismissal | Start here, Variants, States, Do/Don't, API | Action affordances, Urgency |

## 5. Premium Docs Page Structure

### Recommended section order

1. **Title** — Component name + status badge (from meta/hero)
2. **Start here** — Short summary + purpose (required)
3. **Primary preview** — Best example (Default story)
4. **When to use / When not to use** — 2–3 bullets each (required)
5. **Usage guidance** — Do/Don't, content guidelines (required)
6. **Variants and states** — AllVariants, AllSizes, AllStates (required)
7. **Composition examples** — InContext stories (optional)
8. **Accessibility** — Notes, keyboard, ARIA (required)
9. **Known limitations** — Constraints, caveats (optional)
10. **API / Props** — ArgsTable (required)
11. **Recipes** — Copy-paste patterns (optional)
12. **Related components** — Links (optional)
13. **Status / migration** — When deprecated or changing (optional)

### Required vs optional

| Section | Required | Populated from |
|---------|----------|----------------|
| Title, Summary | Yes | meta.title, description.component |
| Start here | Yes | description.component first para or docs.startHere |
| Primary preview | Yes | Default story |
| When to use | Yes | description.component or docs.whenToUse |
| Usage (Do/Don't) | Yes | docs.usage |
| Variants/States | Yes | Stories |
| A11y | Yes | docs.accessibility |
| API | Yes | ArgTypes |
| Limitations | No | docs.limitations |
| Recipes | No | docs.recipes |
| Related | No | docs.related |

### Patterns

- **Start-here block**: Single paragraph or short list; `docs.startHere` or first `##` of description.
- **Do/Don't**: UsagePanel (Admin) or DocsCallout (Miniapp); `docs.usage.doItems` / `dontItems`.
- **State matrix**: Optional table (story name → states) for complex components.
- **Known limitations**: Bullet list; `docs.limitations: string[]`.
- **Recipe block**: Title + code block + 1-line description; `docs.recipes: { title, code, description }[]`.
- **Related components**: Link list; `docs.related: { label, href }[]`.

## 6. Visual Design Rules

| Rule | Value | Rationale |
|------|-------|-----------|
| Page max width | 72rem (1152px) | Comfortable reading; not full-width |
| Content width (prose) | 65ch (optional) | Optimal line length |
| Section gap | `--ds-docs-section-gap` (var(--space-12)) | Clear rhythm |
| H1 | Display font, 24–28px | Clear hierarchy |
| H2 | Left border accent, 20px | Section breaks |
| H3/H4 | 17px / 15px | Sub-sections |
| Body | 14–16px, line-height 1.5–1.6 | Readability |
| Table cell padding | `--ds-table-cell-padding` | Reduced density |
| Code block | `--font-mono`, 0.9rem, line-height 1.6 | Readability |
| Preview frame | Border, padding, "Preview" label | Curated demo |
| Callout | Left border, subtle bg, no icon overload | Professional |
| Zebra striping | Light alternate rows | Table scanability |

## 8. Implementation Strategy

| Mechanism | Use for |
|-----------|---------|
| **DocsContainer** | Theme, docs.page override |
| **Theming** | Manager + docs colors, fonts |
| **parameters.docs** | `startHere`, `usage`, `limitations`, `related`, `recipes`, `sections`, `hero` |
| **parameters.docs.components** | Callout, custom blocks |
| **Decorators** | PreviewFrame, context shell, responsive viewport |
| **Globals/toolbars** | Theme, viewport, platform (Miniapp) |
| **Docs-only stories** | Intro, IA overview, when useful |
| **Custom doc block composition** | MDX only when higher value than default |

### Auto-extraction strategy

- When `parameters.docs.propsTable.rows` is absent, derive rows from `argTypes` (category, type, default, description).
- Use `argTypes[].table.category` for grouping; dim or collapse "Advanced" by default.
- Fallback: Admin shows "Add parameters.docs.propsTable.rows or use argTypes" if neither present.

## 11. Governance Rules

### Every component docs page must include

- Title + status badge
- One-line summary (start here)
- Primary preview (Default story)
- When to use / when not to use (or in description)
- Do/Don't (docs.usage)
- Variants and states (stories)
- Accessibility notes (docs.accessibility)
- API (ArgsTable or PropsTable)

### Optional

- Known limitations
- Recipes
- Related components
- Composition examples
- Migration notes

### Avoid

- Prop dump with no guidance
- Lorem ipsum or fake toy data
- Identical docs structure for every component (vary by category)
- Manual MDX for every component (use MDX only when higher value)
- Exposing every prop with no curation (use argTypes exclude/category)

### When to use full MDX/manual docs

- Overview pages (Foundations, Primitives, Patterns)
- Complex multi-component flows
- Design system governance pages
- When autodocs composition cannot achieve the desired flow

### Naming and section consistency

- Use `##` headings in description; consistent order per STORY-DOCUMENTATION-LAYERS.
- Story names: Default, AllVariants, AllSizes, AllStates, InContext, EdgeCases, Playground, etc.
- argTypes category: Appearance, Behavior, Content, Accessibility, Advanced.

## 12. Final Recommended Target Standard

The finished Autodocs experience should:

1. **Be a decision tool** — Clear "use when" / "don't use when" / "use X instead of Y."
2. **Be a design reference** — Variants, states, composition, content guidelines.
3. **Be a QA surface** — Realistic demos, edge cases, accessibility expectations.
4. **Be a usage guide** — Start-here summary, recipes, related components, known limitations.
5. **Be a polished internal product** — Intentional layout, spacing, typography, framed previews, scannable tables.

**Concrete criteria**:

- Every component page has: title, status, summary, primary preview, when to use, Do/Don't, variants/states, a11y, API.
- Optional sections (limitations, recipes, related) appear when `parameters.docs.*` is set.
- ArgsTable is auto-extracted or manual; grouped by category; Advanced dimmed or collapsed.
- Preview areas are framed with label; code blocks have language label and copy button.
- Callouts support info, tip, warning, danger, success.
- No page feels like a raw prop catalog; each feels curated and component-aware.
- Designers, engineers, QA, and product can find what they need in under 30 seconds.
