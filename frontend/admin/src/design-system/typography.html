Deployment Architecture

The platform uses a rolling deployment strategy to update clusters without service interruption. Each revision is validated against a pre-flight checklist before proceeding to the rollout phase.

Rollout Strategy

Nodes are updated in batches of 25% by default. The orchestrator waits for each batch to pass health checks before continuing. If any node fails to become healthy within 90 seconds, the deployment is automatically paused.

// Rollout configuration
{
  strategy: "rolling",
  batchSize: 0.25,
  healthCheckTimeout: 90,
  onFailure: "pause"
}

Rollback

A rollback window of 15 minutes is available after each deployment. During this window, the previous revision is still active on at least one node. After the window closes, the old revision is fully decommissioned.

Never deploy during a maintenance window without explicit approval from the on-call engineer. Use the #deploys channel to announce intent.

Health Checks

Each node exposes a /healthz endpoint that returns system status. The orchestrator polls this endpoint every 10 seconds. Three consecutive failures trigger a failover event.

  • HTTP 200 — node healthy, traffic accepted
  • HTTP 503 — node degraded, traffic reduced
  • Timeout — node unreachable, failover initiated
